1.1 课题背景与意义
随着移动互联网技术与智能终端设备的迅速普及，基于移动平台的信息获取与学习方式已成为人们日常生活的重要组成部分 。如何利用信息技术手段对中华优秀传统文化进行系统化整理、传播与创新表达，已成为当前文化传承与数字化建设中的重要课题 。 
传统茶道文化作为中华传统文化的重要组成部分，蕴含着深厚的历史底蕴、人文精神与审美价值 。但在现代快节奏的生活中，尤其是在青年群体中，茶道文化的传播方式仍相对单一，系统化的学习与交流渠道较为匮乏 。目前，茶道学习主要依赖书籍或零散的网络资讯，存在学习成本高、内容碎片化、互动性不足等问题 。 
本课题通过开发“传统茶道文化学习与交流APP”，不仅能够为用户提供系统、规范的茶文化学习资源，还将移动开发技术与人工智能检索技术相结合，旨在降低学习门槛，提升茶道文化的传播效率 。这不仅能为用户提供系统、规范的茶文化资源，还能通过社区互动与智能问答增强用户参与感，对传统文化的现代化传播具有重要的现实意义 。 
1.2 国内外研究现状
在国内，随着“数字文化”“文化自信”等理念的提出，学者们开始关注传统文化与信息技术融合的问题。针对茶文化的数字化传播已积累了一定成果。典型的平台如“茶语网”和“茶友汇”，它们在茶叶知识普及、茶艺视频教学和社区交流方面起到了积极作用 。然而，调研发现现有平台仍存在明显短板：一是功能模块较为分散，难以形成从“学习”到“服务”的完整闭环；二是智能化程度不足，用户在面对海量专业知识时，往往难以快速获得精准的个性化解答 。
在国外，虽然缺乏直接以“中国茶道文化”为核心的应用研究，但在文化遗产数字化、移动学习平台设计以及社区化学习系统等方面已有较为成熟的研究成果。相关研究强调通过信息技术手段对传统文化进行数字化保存和传播，并通过交互式设计增强用户体验。
1.3 课题主要研究内容
本课题主要围绕茶文化平台的前后端开发与系统实现展开，重点完成了后端接口设计、数据管理、业务功能开发以及 uni-app 前端页面构建，使用户能够在统一平台上实现茶文化内容浏览、社区互动、活动报名、商品展示等常规功能。在此基础上，课题进一步加入了基于向量检索和大模型的问答能力作为扩展功能，用于提升茶文化知识查询的智能化水平，但这部分并不是本课题的主要研究重点。整体上，本课题通过前后端协同开发与功能集成，形成了一个结构完整、可运行的茶文化综合应用系统。
2  相关技术综述
2.1 开发平台及环境简介
本项目的开发与运行环境以常见的 Web 全栈技术栈为主：后端基于 Node.js 生态，数据库采用 MySQL 做持久化存储，AI 向量服务采用 Python + FastAPI 以便快速提供嵌入/推理接口。前端使用 uni-app + Vue 3 框架，通过 Vite 实现快速的本地开发与构建。开发环境推荐使用支持 Node.js 与 Python 的现代操作系统，并在本地或容器中配置相应的虚拟环境与依赖锁定，以保证可复现性。
2.2 前端开发技术（uni-app、Vue 3、Vite）
前端采用 uni-app 实现跨平台（H5、小程序、App）页面，核心视图层使用 Vue 3 的 Composition API 编写组件、管理状态与生命周期。Vite 提供了极速启动与按需热重载，提升开发效率并且生成的产物便于在多平台中集成。项目中还使用统一的 HTTP 请求封装、路由与状态管理模块，以保证页面与后端接口的一致交互逻辑。
2.3 后端及数据库技术（Node.js、Express、MySQL）
后端以 Node.js 为运行时，通过轻量级 Web 框架（如 Express 或类似中间件风格的实现）构建 RESTful API，负责用户认证、内容管理、活动与订单等业务逻辑。MySQL 用于结构化数据存储与事务处理，配合连接池与索引优化以满足并发请求性能需求。后端还包含与 AI 向量服务的集成层，用于调用嵌入接口或检索结果并向前端返回聚合数据。
2.4 AI 相关技术（FastAPI、RAG 架构、向量检索）
AI 部分以 Python 实现的微服务方式部署（推荐使用 FastAPI 提供 HTTP 接口），负责模型加载、文本向量化与相似度检索。本项目将向量检索作为可选扩展功能：将文本切分并使用中文 embedding 模型生成向量，通过余弦相似度或近似最近邻检索召回相关片段，再把召回的上下文与用户问题一并送入大语言模型生成答案（即检索增强生成，RAG）。该架构有利于把大模型的生成能力与领域知识检索结合，同时保留本地知识回退策略以保证稳定性。

3  系统分析
3.1 可行性分析
3.1.1 技术可行性
本项目采用的技术栈均为当前业界主流且经过验证的开源技术框架。在后端架构方面，选用 Node.js 结合 Express 框架来构建 RESTful API，该框架轻量高效、生态完善，已在众多大型项目中得到广泛应用。前端跨平台开发则采用 uni-app + Vue 3 的组合方案，可实现一套代码编译部署至 H5、小程序、原生 App 等多个平台，显著降低了多平台开发的成本与复杂度。数据持久化层选择 MySQL 关系型数据库，其成熟稳定的事务处理与数据一致性保证能够满足业务需求。AI 服务模块采用 Python 的 FastAPI 框架，该框架以现代高性能著称，支持异步编程与自动化 API 文档生成，便于快速迭代与集成。向量检索与大模型的集成则基于开源的中文 embedding 模型（如 BERT 系列）和商业的大语言模型 API 服务（如 OpenAI、阿里通义千问等），这些都具有成熟的文档与社区支持，可快速集成到系统中。

3.1.2 经济可行性
从开发成本角度看，本项目采用的开源技术框架不产生任何许可费用，大幅降低了前期投入。服务器基础设施可选择部署在阿里云、腾讯云等云计算平台上，初期可采用按量付费的方式控制成本，随着业务增长后再逐步扩容，避免过度投资。AI 模型的调用费用通过选择合理的商业模式（如面向普通用户免费体验、会员用户高级功能付费、企业客户订阅等）可得到有效控制，并形成多元化的变现渠道。内容层面，平台可通过与茶文化研究机构、茶艺师、茶商等建立合作关系获取高质量内容，同时也支持用户生成内容（UGC）模式，进一步降低内容获取成本。综合来看，该项目具备较强的商业化潜力，通过会员订阅、内容变现、商品销售等多种方式可实现盈利目标。

3.1.3 风险评估与应对
项目开发与运营过程中存在若干风险需要提前识别与防范。内容合规风险方面，由于平台支持用户生成内容，可能出现违法违规信息，需建立多层次的审核机制，包括自动化内容过滤（利用文本分类模型）与人工审核相结合，对违规内容及时下架。数据安全风险涉及用户隐私与交易信息的保护，应采取 HTTPS 加密传输、密码加盐哈希存储、定期的安全审计与渗透测试等措施。系统高并发访问时可能出现性能瓶颈，通过引入 Redis 缓存、数据库查询优化与索引设计、CDN 加速等手段可有效提升系统承载能力。AI 问答模块的模型精度可能不稳定，应建立用户反馈机制，持续迭代优化模型效果，同时设置置信度阈值，对低置信度的问题可转交人工客服处理，保证用户体验。

3.2 需求分析
3.2.1 功能需求
系统围绕茶文化学习、社区交互、商业应用等多个维度展开功能设计。用户管理与认证是基础功能，系统需要支持普通用户、内容创作者与平台管理员三种角色。在认证方面，支持手机号与邮箱两种注册方式，采用 JWT（JSON Web Token）机制实现无状态会话认证，确保跨客户端的兼容性。每个用户拥有独立的个人资料页面，包括头像、签名、关注列表等信息。权限管理上，不同角色的用户享有差异化的操作权限，普通用户主要进行浏览与参与互动，内容创作者可发布原创内容，管理员掌控整个平台的运营与维护。账户安全涉及密码修改、登出管理、异常登录告警等功能。

茶文化内容管理是平台的核心功能模块。内容组织采用多维度分类方式，包括按茶类（绿茶、白茶、黄茶等）、按茶艺技能等级、按茶道历史时期等维度分类，支持用户精准检索所需内容。内容形式上支持富文本、图片、视频等多媒体格式，其中视频资源可通过嵌入或外链的方式集成。为了提升用户体验，系统提供基于全文搜索的内容查询能力，同时支持按分类、热度、时间等多种排序方式筛选。用户对内容的反馈通过评分与评论机制记录，这些反馈数据有助于内容质量评估与推荐算法优化。

社区交互功能旨在增强用户粘性并促进知识交流。用户可在社区发布图文动态、提出茶文化相关的提问、分享个人的茶道心得与经验，充分激发社区活力。评论与点赞机制使得优质内容能够获得更多曝光，形成激励机制。茶友社区功能允许用户基于兴趣标签建立群组，便于志同道合的茶艺爱好者聚集交流。系统还需提供消息通知机制，当用户的内容获得评论、点赞、转发时及时推送提醒，保持用户活跃度。此外，用户可关注感兴趣的创作者或用户，方便获取其最新发布的内容。

活动与订单管理模块支撑平台的商业闭环。管理员可在平台发布线下茶艺表演、品茶会、茶道讲座等活动信息，用户可在线报名参与。系统需记录活动的签到信息，并在活动结束后收集用户的满意度反馈。商品管理功能支持上传茶叶、茶具、茶书等相关产品，包括商品分类、定价、库存管理等。用户通过购物车功能选择商品，完成订单创建与支付流程。订单生成后，用户可实时追踪订单状态（待支付、已发货、已收货等），同时系统展示物流信息。售后环节支持用户提交退换货申请，由平台审核与处理。

智能问答系统作为平台的特色功能，采用检索增强生成（Retrieval-Augmented Generation, RAG）的架构。用户向 AI 助手提出关于茶文化的自由提问，系统首先对问题进行向量编码，与预先建立的茶文化知识库进行向量相似度匹配，召回最相关的知识片段作为上下文。这些知识片段与原始问题一并输入大语言模型，生成融合领域知识的答案。系统支持多轮对话交互，用户可持续追问以获得更深入的理解。为确保答案质量，系统提供用户评价机制，用户可点赞或反馈不满意的答案，这些反馈数据用于后续模型微调与算法优化。

3.2.2 非功能需求
在性能方面，系统支持基本的查询请求处理，响应时间取决于网络条件与后端服务的可用性。内容搜索通过向量检索与全文匹配相结合实现，AI 问答则依赖向量检索召回与大语言模型的推理，完整流程涉及网络延迟与外部服务响应时间。系统在本地开发环境可同时处理多个并发请求，生产部署可根据实际负载通过横向扩展。

安全性需求主要包括身份认证与访问控制。用户密码采用 bcrypt 加密存储（盐轮次为 10），使用 JWT（JSON Web Token）实现无状态会话认证，确保跨客户端的兼容性。系统基于角色（user/admin）实现权限区分，普通用户只能访问与修改自己的数据，管理员拥有平台级的管理权限。内容审核方面，系统提供自动化审核接口，对用户发布的内容进行合规性检测，标记出存在风险的内容供人工审核处理。

可扩展性设计采用模块化架构。系统将用户管理、内容管理、社区功能、订单系统、AI 服务等解耦为独立的模块与服务，各部分通过统一的 RESTful 接口通信，便于后续功能扩展与独立维护。前端采用 uni-app 框架实现跨平台支持，后端基于 Express 框架运行，数据库采用 MySQL 关系型存储。系统各模块可独立部署与升级，当需要扩展时可单独增强特定模块（如增加数据库实例、部署多个后端服务实例等），提高系统的灵活性与可维护性。

3.3 系统流程分析
3.3.1 用户业务流程
用户在平台上的交互流程涵盖账户创建、内容浏览、社区参与、订单交易等多个环节，形成一个完整的使用闭环。新用户首先需注册账户，选择手机号或邮箱方式，设置强密码后即可创建账户。账户创建后，用户进入首页，系统基于其浏览历史与兴趣标签推荐相关的茶文化内容，同时提供分类浏览功能让用户主动探索。用户可在学习版块深入阅读内容，也可在社区版块发布自己的见解、提出问题或分享经验，与其他用户进行互动与讨论。当用户发现感兴趣的商品时，可加入购物车，在确认订单信息与收货地址后完成支付。订单支付成功后，用户可在个人中心查看订单物流信息，待商品送达并确认收货后可对商品进行评价。此外，用户在平台使用过程中任何时刻都可进入 AI 对话模块，向智能助手提出关于茶文化的问题，获得实时的智能化回答。

3.3.2 内容管理流程
平台内容的完整生命周期从创建经历审核、发布、互动、反馈等多个阶段。内容创作者（包括平台编辑与认证用户）在后台编辑器中撰写茶文化相关的内容，包括图文与视频等形式，输入内容分类、标签等元信息后提交发布。系统后台自动触发审核流程，首先通过内容安全模型进行自动检测，对文本、图片进行合规性扫描，标记出存在风险的内容。自动审核通过的内容可立即发布至前台供用户浏览，被标记为风险的内容则进入人工审核队列，由运营人员逐一审查。发布后的内容在平台上获得用户浏览、评论、评分等互动，这些数据被记录用于后续的内容评估。平台运营人员定期分析内容的互动数据与用户反馈，对违规内容进行下架处理，对高质量且高热度的内容进行特殊推荐，进一步提升平台的内容质量与用户体验。

3.3.3 AI 问答流程
用户与 AI 助手的交互遵循一个精心设计的处理流程，确保答案的准确性与相关性。用户在 AI 对话模块输入关于茶文化的自然语言问题后，系统首先对问题进行文本预处理，包括分词、去停用词、纠错等操作，随后利用训练好的中文 embedding 模型将问题转换为向量表示。系统将问题向量与知识库中预先计算好的所有内容向量进行相似度计算（通常采用余弦相似度指标），从中召回与问题最相关的 Top-K 个知识片段作为上下文。这些知识片段、原始问题以及必要的系统提示词被组织成提示（prompt），输入到大语言模型中。大语言模型基于所提供的上下文与问题生成最终答案，确保答案既准确又贴切用户的具体需求。答案返回给用户前，系统进行质量检查，包括合规性审核（确保答案不包含违规内容）与置信度评估（确保模型的生成可信度达到阈值）。用户收到答案后可进行评价，包括点赞、点踩或提交详细反馈，这些评价数据被收集并用于后续的模型微调与算法优化，形成持续改进的反馈闭环。

4.3 前端设计
4.3.1 页面结构与导航
前端页面按照“首页、学习、社区、我的”四个主入口组织，核心导航由底部 tabBar 承担，对应 `pages/index/index`、`pages/learning/learning`、`pages/community/community` 和 `pages/profile/profile`。围绕主入口系统还提供了登录、内容详情、帖子详情、发布动态、课程详情、订单详情、活动列表、报名页面、意见反馈、AI 问答以及管理员后台等辅助页面，形成完整的业务闭环。首页用于内容推荐与快捷入口，学习页用于知识浏览和课程查看，社区页用于帖子浏览和互动，个人中心用于用户资料、订单、发布记录和反馈提交。

4.3.2 组件设计与状态管理
前端采用 uni-app 进行跨端开发，并通过 easycom 自动注册 `uview-plus` 组件以减少重复引入。页面之间共享的登录状态、用户信息和令牌信息由全局状态模块统一维护，`store/index.js` 负责导出用户状态仓库，便于各页面直接读取登录态。网络请求统一通过 `api/index.js` 与 `utils/http.js` 封装，页面只需关注参数与返回结果即可完成接口调用，这样可以降低页面和后端接口之间的耦合度。

4.4 后端接口设计
4.4.1 API 设计规范
后端接口采用 RESTful 风格设计，按照资源维度组织路由，常见方法包括 `GET`、`POST`、`PUT`、`PATCH` 和 `DELETE`。公开接口主要用于内容浏览、详情查询和搜索；需要登录的接口通过 JWT 令牌进行鉴权；管理员接口则在鉴权基础上进一步校验角色权限。接口返回值统一采用 JSON 格式，便于前端根据 `code`、`message` 和业务数据进行状态处理。

4.4.2 核心接口列表
认证接口包括用户注册、登录和获取当前用户信息；内容接口包括内容列表、内容详情、新增、更新和删除；社区接口包括帖子列表、帖子详情、发帖、评论列表、评论发布与删除；商品接口包括商品列表与商品详情；活动接口包括活动列表、活动详情、活动创建、更新、删除以及报名管理；课程接口包括认证课程列表、详情、创建、更新、删除以及报名管理；订单接口包括创建订单、订单列表、订单详情和取消订单；反馈接口包括反馈提交、反馈列表、反馈详情与处理；AI 接口则包括茶叶知识搜索与智能问答。上述接口覆盖了当前系统的主要业务场景。

4.5 AI 服务设计
4.5.1 向量化与检索流程
AI 服务由独立的 Python 微服务承担，负责将茶文化知识文本转换为向量并提供嵌入接口。系统先对知识文本进行切分和整理，再调用嵌入服务生成向量，最终将向量结果保存到 `tea_with_vectors.json` 等文件中。后续查询时，后端会读取这些向量数据并计算相似度，从而召回与问题最相关的知识片段。

4.5.2 RAG 集成方案
RAG 流程由“检索 + 生成”两步构成。后端在收到用户问题后，先调用检索服务获取 Top-K 相关片段，再将问题与片段拼接成上下文传递给大模型生成答案。系统在大模型不可用时会返回本地回退答案，保证问答接口的连续性。该设计将领域知识检索与大模型生成能力结合起来，既提高了答案相关性，也增强了系统的可用性与扩展性。
